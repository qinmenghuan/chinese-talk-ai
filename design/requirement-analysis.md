# 中文口语练习 AI 网站需求分析

## 1. 文档目的
基于 [reqirement.md](D:/study-program/AI/learn_chinese/design/reqirement.md) 中的原始描述，对项目目标、范围、实现路径和风险进行结构化分析，形成可执行的需求分析文档。

## 2. 项目概述
该项目拟开发一个面向中文口语练习的 AI 聊天网站，支持中文为主、英文为辅的实时互动。结合“练习中文口语”“参考豆包实时聊天”“实时显示聊天记录”等描述，本文将项目定义为一个“语音优先，文本同步展示”的 AI 语言练习应用。

产品的核心价值不是通用聊天，而是帮助用户通过实时对话练习中文表达，并在练习结束后得到一份针对表达内容、用词、语法和流利度的中文分析报告。

## 3. 目标与价值
### 3.1 用户目标
- 与 AI 进行实时对话，降低开口练习中文的门槛。
- 在对话过程中实时看到聊天记录，便于理解、跟读和复盘。
- 在练习结束后获得一份结构化分析报告，用于持续改进中文口语能力。

### 3.2 产品价值
- 为用户提供低压力、可重复的口语练习场景。
- 通过实时转写与实时回复增强对话沉浸感。
- 通过会话复盘和分析报告形成“练习 -> 反馈 -> 再练习”的闭环。

## 4. 需求理解与范围界定
### 4.1 已明确需求
- 网站支持与 AI 实时聊天。
- 主要支持中文，同时兼容英文聊天。
- 页面中实时展示聊天记录。
- 练习结束后保存或记录聊天内容。
- 基于聊天记录生成中文分析报告。
- 优先先做核心能力，再逐步优化细节。

### 4.2 隐含需求
从“练习中文口语”和“参考豆包实时聊天”可以推断出以下隐含需求：
- 产品核心交互不是纯文本聊天，而是完整的语音对话体验。
- 对话体验需要低延迟，不能是“说完很久后才回复”。
- 需要有会话开始、进行中、结束和复盘几个明确阶段。
- 在不登录的前提下，仍需要匿名会话标识来管理聊天记录和报告。

### 4.3 已确认的产品约束
- MVP 必须支持“语音输入 + 语音输出”。
- MVP 不需要登录体系，允许匿名体验。
- 数据存储层面只保存转写文本，不保存原始音频。
- 报告输出需要覆盖语法、词汇、流利度、发音、地道程度等评价维度。
- 中文口语练习在 MVP 阶段即需要分场景，至少包含日常聊天、面试、旅游、商务。

## 5. 目标用户与典型场景
### 5.1 目标用户
- 主要目标用户是海外中文学习者，尤其是需要高频口语练习的学生群体。
- 希望通过 AI 陪练进行日常表达训练的学习者。
- 需要低成本、随时可练习的个人用户。

### 5.2 典型使用场景
- 用户打开网页，选择练习场景和中文练习模式，点击开始对话。
- 用户通过麦克风说中文，系统实时转写并展示到聊天区。
- AI 以语音为主、文字同步展示的方式实时回复，用户继续对话。
- 对话结束后，系统生成一份中文分析报告，指出优点和改进项。
- 用户查看历史练习记录，进行复盘。

## 6. 功能需求拆解
### 6.1 MVP 必做功能
#### 1. 实时会话
- 支持用户发起一轮实时对话。
- 支持用户选择练习场景，如日常聊天、面试、旅游、商务。
- 支持文本消息流式展示。
- 支持 AI 语音流式回复，并同步展示文本。
- 支持基础的中英文识别与回复。

#### 2. 聊天记录展示
- 用户发言和 AI 回复按时间顺序展示。
- 正在识别中的内容支持增量更新。
- AI 正在生成中的回复支持流式追加。

#### 3. 匿名会话与结束保存
- 用户无需登录即可开始体验。
- 系统为每次练习生成匿名会话标识。
- 用户可手动结束练习。
- 系统保存本次会话的文本记录。
- 会话具备唯一 ID，便于后续查看报告。

#### 4. 分析报告
- 根据完整聊天记录生成中文分析报告。
- 报告至少包含：总体评价、表达优点、常见问题、改进建议、建议练习方向。
- 报告需要覆盖语法、词汇、流利度、发音、地道程度五个维度。

### 6.2 第二阶段建议功能
- 发音纠错与更细颗粒度评分。
- 扩充场景化练习模板，如课堂交流、商务谈判、客户沟通。
- 历史记录列表、搜索和标签。
- 用户登录、练习统计、连续打卡。
- 单词或表达高亮，支持点击查看解释。

### 6.3 非功能需求
- 首屏进入快，开始会话步骤少。
- 会话延迟可接受，实时体验稳定。
- 聊天内容存储安全，敏感数据可控。
- 断网、权限拒绝、识别失败时能给出明确提示。
- 后端日志和监控可定位实时链路问题。

## 7. 用户流程
### 7.1 主流程
1. 用户进入首页。
2. 用户选择练习场景。
3. 用户允许麦克风权限。
4. 用户开始实时对话。
5. 前端实时展示用户语音转写文本。
6. AI 返回流式语音回复，并同步展示文本。
7. 用户多轮练习。
8. 用户结束会话。
9. 后端保存会话记录与分析结果。
10. 系统生成并展示中文分析报告。

### 7.2 异常流程
- 麦克风权限被拒绝：提示用户开启权限；由于 MVP 以语音为核心，不提供纯文本替代主流程。
- 网络波动：提示重连或自动恢复会话。
- AI 接口超时：提示稍后重试，并保留已完成的聊天记录。
- 报告生成失败：先保存会话，报告异步补生成。

## 8. 可行性分析
### 8.1 业务可行性
该项目聚焦单一场景，即中文口语练习，问题边界清晰，适合个人开发者以 MVP 方式快速验证。由于主要目标用户是海外中文学习者，产品在业务上应优先考虑英文引导、海外访问稳定性、国际支付与全球部署便利性。核心价值明确，且能用较少页面承载主要功能，因此业务上可行。

### 8.2 技术可行性
技术上可行，原因如下：
- 你已有 React、Next.js、NestJS、PostgreSQL、Redis、Docker 经验，能够覆盖前后端、存储和部署主链路。
- 当前主流 AI 平台已提供实时语音、实时转写、流式文本和会话式推理能力，不需要从零搭建语音识别或大模型服务。
- 实时聊天的难点主要在音频链路、延迟控制和状态同步，但这些属于工程复杂度问题，不是不可实现的问题。

### 8.3 产品实现策略可行性
建议采取“先打通语音闭环，再优化评估细节”的策略：
- 第一阶段：先做语音输入、语音输出、实时转写展示、匿名会话保存。
- 第二阶段：补全多场景、结构化分析报告、历史记录和学习统计。

这样的拆分能显著降低首版开发风险。

## 9. 实现成本与主要难点
### 9.1 成本构成
#### 1. 开发成本
- 前端实时音频采集、状态管理和聊天 UI。
- 后端会话管理、记录存储、报告生成、第三方 API 封装。
- 数据库表设计和基础后台能力。

#### 2. 第三方服务成本
- 实时 AI 会话费用。
- 语音识别或语音合成费用。
- 数据库存储、对象存储、云部署费用。

#### 3. 维护成本
- 接口稳定性维护。
- 提示词和报告质量迭代。
- 音频链路调优与错误排查。

### 9.2 官方价格口径下的成本估算
以下估算以 2026 年 5 月 21 日可查到的官方价格页面为准。由于 OpenAI 使用美元计价、火山引擎使用人民币计价，本文不做汇率换算，避免因汇率波动导致误差。

#### 1. OpenAI Realtime API 成本估算
适用前提：
- 面向海外中文学习者，优先保证海外访问稳定性与开发效率。
- 使用 `gpt-realtime-2` 完成语音输入和语音输出。
- 使用 `gpt-realtime-whisper` 做实时转写，以满足“界面实时显示聊天记录”的需求。

官方价格口径：
- `gpt-realtime-2` 音频输入：32 美元 / 100 万 token
- `gpt-realtime-2` 音频输出：64 美元 / 100 万 token
- `gpt-realtime-whisper`：0.017 美元 / 分钟

基于官方成本文档中的 token 换算逻辑：
- 用户输入音频约为每 100ms 1 个 token，即约 600 token / 分钟。
- AI 输出音频约为每 50ms 1 个 token，即约 1200 token / 分钟。

按单分钟粗略估算：
- 用户连续说 1 分钟，输入音频成本约 0.0192 美元 / 分钟。
- AI 连续说 1 分钟，输出音频成本约 0.0768 美元 / 分钟。
- 若开启实时转写，用户语音转写成本约 0.017 美元 / 分钟。

按一次 10 分钟练习估算：
- 假设用户累计说 5 分钟，AI 累计说 5 分钟。
- 音频输入成本约 0.096 美元。
- 音频输出成本约 0.384 美元。
- 实时转写成本约 0.085 美元。
- 小计约 0.565 美元 / 次。

预算建议：
- 如果再计入文本 token、上下文增长和缓存未命中开销，建议按 0.65 到 0.75 美元 / 10 分钟会话做预算更稳妥。
- 如果每月有 1000 次 10 分钟练习，会话模型成本大致可按 650 到 750 美元 / 月预估。

#### 2. 火山引擎豆包语音成本估算
适用前提：
- 使用“豆包端到端实时语音大模型”完成语音对话。
- 仅保存文本，不保存原始音频。
- 主要用于对比国内语音方案成本。

官方价格口径：
- 输入文本：10 元 / 100 万 token
- 输入音频：80 元 / 100 万 token
- 输入文本 cached：5 元 / 100 万 token
- 输入音频 cached：5 元 / 100 万 token
- 输出文本：80 元 / 100 万 token
- 输出音频：300 元 / 100 万 token

基于官方文档中的 token 折算逻辑：
- 当前轮用户输入音频约 6.25 token / 秒，即约 375 token / 分钟。
- 其他输入音频或输出音频约 25 token / 秒，即约 1500 token / 分钟。

按单分钟粗略估算：
- 用户连续说 1 分钟，输入音频成本约 0.03 元 / 分钟。
- AI 连续说 1 分钟，输出音频成本约 0.45 元 / 分钟。

按一次 10 分钟练习估算：
- 假设用户累计说 5 分钟，AI 累计说 5 分钟。
- 音频输入成本约 0.15 元。
- 音频输出成本约 2.25 元。
- 小计约 2.40 元 / 次。

预算建议：
- 若再计入文本 token、上下文和缓存变化，建议按 2.5 到 2.8 元 / 10 分钟会话做预算。
- 如果每月有 1000 次 10 分钟练习，会话模型成本大致可按 2500 到 2800 元 / 月预估。

#### 3. 其他开发与运行费用
除模型费用外，还应预留以下成本：
- 域名：通常约 10 到 20 美元 / 年。
- 前端托管：Vercel Pro 为 20 美元 / 月，包含 20 美元用量额度；Hobby 免费，但更适合个人非商业试运行。
- 后端托管：
  - Railway Hobby 为 5 美元最低消费，含 5 美元额度。
  - Railway Pro 为 20 美元最低消费，含 20 美元额度。
  - Render Pro 为 25 美元 / 月，另加计算资源费用；其 Web Service `Starter` 为 7 美元 / 月起。
- 数据库与缓存：
  - Render Postgres 可从 6 美元 / 月起。
  - Render Key Value 可从 0 美元 / 月起。
  - Upstash Redis 免费版为 0 美元，按量付费为 0.2 美元 / 10 万命令。
- 日志与监控：开发阶段可先用平台自带日志，生产阶段通常还会新增 0 到 20 美元 / 月不等的监控成本。
- 邮件通知、埋点、错误上报：首版可以控制在免费额度内，但正式上线后通常会产生小额月费。

#### 4. 首版项目月成本建议
如果采用“Vercel + Railway + OpenAI”的海外优先方案，首版常见月成本可按以下思路预估：
- 固定基础设施成本约 25 到 40 美元 / 月。
- 模型成本随使用量线性增长，通常会远高于服务器本身。
- 因此首版预算重点不在服务器，而在控制单次会话时长、轮数和模型调用密度。

### 9.3 主要技术难点
#### 1. 实时链路复杂
实时语音聊天涉及浏览器录音、传输协议、服务端会话、AI 流式响应、前端同步渲染，链路比普通聊天长，调试成本高。

#### 2. 延迟与体验平衡
口语练习对停顿和响应速度敏感。识别、推理和合成任何一段过慢，都会影响“像在对话”的体验。

#### 3. 转写与展示一致性
实时转写往往先输出临时文本，再输出最终文本，前端需要处理增量修正，否则会出现重复、跳字、顺序错乱等问题。

#### 4. 报告质量控制
仅靠一次 prompt 生成的报告容易空泛。若要让报告真正有学习价值，需要设计固定的评价维度和输出模板。

#### 5. 数据模型设计
会话、消息、转写片段、报告之间需要明确关联，否则后续很难扩展历史记录、统计分析和场景练习。

### 9.4 个人开发下的风险评估
对个人开发者来说，最大风险不是代码量，而是同时处理前端音频、后端状态、AI 接口和体验调优。建议强制控制范围，避免首版同时做：登录、支付、逐词级发音评分、过多场景扩展、多端同步等扩展项。

## 10. 建议技术架构
## 10.1 架构原则
- 使用 Monorepo，统一管理前端、后端、共享类型和配置。
- 先保证核心链路跑通，再做复杂拆分。
- 尽量让实时连接直达 AI 服务，应用后端负责鉴权、会话记录和业务编排。

## 10.2 推荐 Monorepo 目录结构
```text
learn_chinese/
  apps/
    web/                 # Next.js 前端
    api/                 # NestJS 后端
  packages/
    ui/                  # 可复用 UI 组件
    types/               # 前后端共享类型
    prompts/             # 报告生成与系统提示词
    eslint-config/       # 共享 lint 配置
    tsconfig/            # 共享 tsconfig
  infra/
    docker/              # Docker 配置
    scripts/             # 部署或初始化脚本
```

## 10.3 前端建议
- 框架：Next.js
- 实时能力：浏览器麦克风采集 + WebRTC 优先，WebSocket 作为服务端流式备选
- 状态管理：Zustand 或 React Context + reducer
- UI：聊天窗口、音频权限提示、会话状态栏、报告页

前端重点处理：
- 麦克风权限
- 扬声器播放和打断控制
- 实时文本增量渲染
- 连接状态和错误提示
- 会话结束后的报告跳转

## 10.4 后端建议
- 框架：NestJS
- 能力边界：
  - 生成实时会话所需的服务端令牌或会话信息
  - 生成匿名会话 ID 或访客标识
  - 管理会话开始、结束、持久化
  - 保存聊天文本、报告结果
  - 保存场景信息与报告维度结果
  - 封装第三方 AI 接口
  - 提供报告生成接口和历史记录接口

## 10.5 数据存储建议
- PostgreSQL：保存匿名会话、消息、报告、场景配置
- Redis：保存实时会话状态、短期缓存、限流和异步任务状态
- 对象存储：MVP 不需要；仅当后续要保存原始音频时再引入

## 10.6 第三方 AI API 建议
### 推荐方案 A：OpenAI 实时语音方案
适合目标：主要服务海外中文学习者，追求较好的实时体验、浏览器端低延迟语音对话、后续可扩展多模态能力。

推荐理由：
- OpenAI 官方文档当前推荐浏览器端语音助手优先使用 Realtime API + WebRTC。
- 官方 Voice agents 文档明确将语言辅导类场景作为适用方向之一。
- 实时会话适合“边说边转写、边听边回复”的体验。

建议用法：
- 实时对话：`gpt-realtime-2`
- 实时转写：`gpt-realtime-whisper`
- 离线高质量转写或复盘：`gpt-4o-transcribe`
- 低成本转写：`gpt-4o-mini-transcribe`

适用实现方式：
- 前端通过 WebRTC 与实时模型建立连接。
- 后端仅负责生成临时会话凭证、保存记录、生成报告和做业务编排。

### 备选方案 B：火山引擎豆包语音方案
适合目标：更贴近中文场景、希望参考豆包式语音体验、优先考虑国内生态或合规落地。

推荐理由：
- 火山引擎官方文档提供豆包端到端实时语音大模型，强调低时延、可打断、自然语音交互。
- 豆包语音也提供流式语音识别能力，适合实时字幕和边说边出文字。
- 对中文口语练习类产品有较强相关性。

适用实现方式：
- 如果首版更强调中文体验和国内可达性，可优先评估豆包实时语音能力。
- 如果首版更强调开发效率和国际通用接口，可优先使用 OpenAI。

### 最终建议
如果你当前目标是尽快做出可演示 MVP，建议优先选择一种平台，不要首版同时接两家。

建议优先级：
1. 如果主要面向海外中文学习者、重视全球可访问性和开发效率，优先 OpenAI。
2. 如果后续重点转向国内市场、希望压低语音链路成本，可再评估火山引擎。

## 10.7 发布平台建议
### 推荐发布方案
针对“海外中文学习者”为主的产品定位，建议采用前后端分离部署：
- 前端发布到 Vercel。
- 后端 API 发布到 Railway 或 Render。
- PostgreSQL 与 Redis 使用托管服务。

推荐原因：
- Vercel 对 Next.js 支持最好，全球 CDN 和预览部署体验成熟，适合面向海外用户的 Web 前端。
- Railway 和 Render 更适合承载 NestJS 常驻 API、定时任务和数据库连接，不必强行把后端塞进前端平台的函数模型中。
- 如果实时语音链路采用前端直连 OpenAI WebRTC，后端负担会较轻，这种部署结构更适合首版。

### 平台选型建议
#### 方案 A：Vercel + Railway
- 适合个人开发者快速上线海外产品。
- 前端放 Vercel，后端放 Railway，部署简单，学习成本低。
- Railway 计费直观，适合从 MVP 到小规模生产的过渡。

#### 方案 B：Vercel + Render
- 适合希望把 Web API、Postgres、Redis 尽量集中在一个平台体系内管理。
- Render 对 Web 服务、数据库和 Key Value 的组合支持更完整。
- 如果后续需要更多后台任务、队列和 WebSocket 扩展，Render 的服务模型更自然。

### 不建议的首版发布方式
- 不建议首版把前端和 NestJS 后端全部压到单一 Serverless 前端平台中。
- 不建议首版自建 VPS 运维整套数据库、Redis、反向代理和监控，这会明显增加个人开发负担。

## 12. MVP 版本定义
建议把 MVP 严格控制为以下范围：
- 单页实时聊天界面
- 场景选择入口
- 麦克风输入
- AI 语音输出
- 实时转写文本展示
- AI 回复文本同步展示
- 匿名体验，无需登录
- 手动结束会话
- 只保存转写文本聊天记录
- 生成一份中文分析报告
- 简单历史记录查看

以下内容不建议放入首版：
- 复杂用户体系
- 支付订阅
- 多角色人设系统
- 发音逐词打分
- 多语言课程体系
- 原始音频长期存储

## 13. 开发阶段计划

目标：跑通核心链路
- 完成 Monorepo 初始化

- 接入一种 AI 实时能力
- 实现语音输入、语音输出和实时聊天文本展示
- 完成匿名会话与文本记录保存

### 第二阶段：1 到 2 周
目标：输出学习闭环
- 完成基础场景配置
- 增加报告生成接口
- 固化报告模板
- 完成报告页和历史页
- 补充异常处理和基础日志

### 第三阶段：持续迭代
目标：提升体验
- 扩充练习场景
- 增加统计与成长反馈
- 优化提示词和延迟体验

## 14. 结论
该项目适合以个人开发方式启动，且具备较高可执行性。关键不在于“能不能做”，而在于“能否把首版范围压到足够小”。

最合理的路径是：
- 先做语音输入 + AI 语音输出 + 文本同步展示的完整闭环
- 再做匿名会话保存和多维度分析报告
- 最后再补更多场景、成长反馈和更精细的评分能力

如果目标用户主要是海外中文学习者，则首版最稳妥的商业与技术组合通常是：
- 模型层优先 OpenAI Realtime API
- 前端优先部署到 Vercel
- 后端优先部署到 Railway 或 Render

只要严格控制 MVP，优先打通实时链路，这个项目可以在较短周期内做出可演示版本。
