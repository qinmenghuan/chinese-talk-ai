# 中文口语练习 AI 网站需求分析

## 1. 文档目的

基于 [reqirement.md](D:/study-program/AI/learn_chinese/design/reqirement.md) 中的原始描述，对项目目标、范围、实现路径和风险进行结构化分析，形成可执行的需求分析文档。

## 2. 项目概述

该项目拟开发一个面向中文口语练习的 AI 聊天网站，支持中文为主、英文为辅的实时互动。结合“练习中文口语”“参考豆包实时聊天”“实时显示聊天记录”等描述，本文将项目定义为一个“语音优先，文本同步展示”的 AI 语言练习应用。

产品的核心价值不是通用聊天，而是帮助用户通过实时对话练习中文表达，并在练习结束后得到一份针对表达内容、用词、语法和流利度的中文分析报告。

## 3. 目标与价值

### 3.1 用户目标

- 与 AI 进行实时对话，降低开口练习中文的门槛。
- 在对话过程中实时看到聊天记录，便于理解、跟读和复盘。
- 在练习结束后获得一份结构化分析报告，用于持续改进中文口语能力。

### 3.2 产品价值

- 为用户提供低压力、可重复的口语练习场景。
- 通过实时转写与实时回复增强对话沉浸感。
- 通过会话复盘和分析报告形成“练习 -> 反馈 -> 再练习”的闭环。

## 4. 需求理解与范围界定

### 4.1 已明确需求

- 网站支持与 AI 实时聊天。
- 主要支持中文，同时兼容英文聊天。
- 页面中实时展示聊天记录。
- 练习结束后保存或记录聊天内容。
- 基于聊天记录生成中文分析报告。
- 优先先做核心能力，再逐步优化细节。

### 4.2 隐含需求

从“练习中文口语”和“参考豆包实时聊天”可以推断出以下隐含需求：

- 产品核心交互不是纯文本聊天，而是完整的语音对话体验。
- 对话体验需要低延迟，不能是“说完很久后才回复”。
- 需要有会话开始、进行中、结束和复盘几个明确阶段。
- 在不登录的前提下，仍需要匿名会话标识来管理聊天记录和报告。

### 4.3 已确认的产品约束

- MVP 必须支持“语音输入 + 语音输出”。
- MVP 不需要登录体系，允许匿名体验。
- 数据存储层面只保存转写文本，不保存原始音频。
- 报告输出需要覆盖语法、词汇、流利度、发音、地道程度等评价维度。
- 中文口语练习在 MVP 阶段即需要分场景，至少包含日常聊天、面试、旅游、商务。

## 5. 目标用户与典型场景

### 5.1 目标用户

- 主要目标用户是海外中文学习者，尤其是需要高频口语练习的学生群体。
- 希望通过 AI 陪练进行日常表达训练的学习者。
- 需要低成本、随时可练习的个人用户。

### 5.2 典型使用场景

- 用户打开网页，选择练习场景和中文练习模式，点击开始对话。
- 用户通过麦克风说中文，系统实时转写并展示到聊天区。
- AI 以语音为主、文字同步展示的方式实时回复，用户继续对话。
- 对话结束后，系统生成一份中文分析报告，指出优点和改进项。
- 用户查看历史练习记录，进行复盘。

## 6. 功能需求拆解

### 6.1 MVP 必做功能

#### 1. 实时会话

- 支持用户发起一轮实时对话。
- 支持用户选择练习场景，如日常聊天、面试、旅游、商务。
- 支持文本消息流式展示。
- 支持 AI 语音流式回复，并同步展示文本。
- 支持基础的中英文识别与回复。

#### 2. 聊天记录展示

- 用户发言和 AI 回复按时间顺序展示。
- 正在识别中的内容支持增量更新。
- AI 正在生成中的回复支持流式追加。

#### 3. 匿名会话与结束保存

- 用户无需登录即可开始体验。
- 系统为每次练习生成匿名会话标识。
- 用户可手动结束练习。
- 系统保存本次会话的文本记录。
- 会话具备唯一 ID，便于后续查看报告。

#### 4. 分析报告

- 根据完整聊天记录生成中文分析报告。
- 报告至少包含：总体评价、表达优点、常见问题、改进建议、建议练习方向。
- 报告需要覆盖语法、词汇、流利度、发音、地道程度五个维度。

### 6.2 第二阶段建议功能

- 发音纠错与更细颗粒度评分。
- 扩充场景化练习模板，如课堂交流、商务谈判、客户沟通。
- 历史记录列表、搜索和标签。
- 用户登录、练习统计、连续打卡。
- 单词或表达高亮，支持点击查看解释。

### 6.3 非功能需求

- 首屏进入快，开始会话步骤少。
- 会话延迟可接受，实时体验稳定。
- 聊天内容存储安全，敏感数据可控。
- 断网、权限拒绝、识别失败时能给出明确提示。
- 后端日志和监控可定位实时链路问题。

## 7. 用户流程

### 7.1 主流程

1. 用户进入首页。
2. 用户选择练习场景。
3. 用户允许麦克风权限。
4. 用户开始实时对话。
5. 前端实时展示用户语音转写文本。
6. AI 返回流式语音回复，并同步展示文本。
7. 用户多轮练习。
8. 用户结束会话。
9. 后端保存会话记录与分析结果。
10. 系统生成并展示中文分析报告。

### 7.2 异常流程

- 麦克风权限被拒绝：提示用户开启权限；由于 MVP 以语音为核心，不提供纯文本替代主流程。
- 网络波动：提示重连或自动恢复会话。
- AI 接口超时：提示稍后重试，并保留已完成的聊天记录。
- 报告生成失败：先保存会话，报告异步补生成。

## 8. 可行性分析

### 8.1 业务可行性

该项目聚焦单一场景，即中文口语练习，问题边界清晰，适合个人开发者以 MVP 方式快速验证。由于主要目标用户是海外中文学习者，产品在业务上应优先考虑英文引导、海外访问稳定性、国际支付与全球部署便利性。核心价值明确，且能用较少页面承载主要功能，因此业务上可行。

### 8.2 技术可行性

技术上可行，原因如下：

- 你已有 React、Next.js、NestJS、PostgreSQL、Redis、Docker 经验，能够覆盖前后端、存储和部署主链路。
- 当前主流 AI 平台已提供实时语音、实时转写、流式文本和会话式推理能力，不需要从零搭建语音识别或大模型服务。
- 实时聊天的难点主要在音频链路、延迟控制和状态同步，但这些属于工程复杂度问题，不是不可实现的问题。

### 8.3 产品实现策略可行性

建议采取“先打通语音闭环，再优化评估细节”的策略：

- 第一阶段：先做语音输入、语音输出、实时转写展示、匿名会话保存。
- 第二阶段：补全多场景、结构化分析报告、历史记录和学习统计。

这样的拆分能显著降低首版开发风险。

## 9. 实现成本与主要难点

### 9.1 成本构成

#### 1. 开发成本

- 前端实时音频采集、状态管理和聊天 UI。
- 后端会话管理、记录存储、报告生成、第三方 API 封装。
- 数据库表设计和基础后台能力。

#### 2. 第三方服务成本

- 实时 AI 会话费用。
- 语音识别或语音合成费用。
- 数据库存储、对象存储、云部署费用。

#### 3. 维护成本

- 接口稳定性维护。
- 提示词和报告质量迭代。
- 音频链路调优与错误排查。

### 9.2 官方价格口径下的成本估算（2026-05-22）

以下估算以 2026 年 5 月 22 日可查到的官方价格页面为准。由于不同平台的计费口径并不完全一致，下面先统一假设，再进行横向比较。

#### 统一估算前提

为了让 OpenAI 和国内方案可比，本文统一采用以下假设：

- 单次练习会话时长：10 分钟。
- 用户有效说话时长：5 分钟。
- AI 有效说话时长：5 分钟。
- AI 输出中文语音对应文本量：约 1100 个汉字。
- 练习结束后生成一份中文分析报告。
- 仅计算满足核心需求所需的模型与语音能力费用，不含前端、数据库、CDN、日志等基础设施费用。

说明：

- 这里的“有效说话时长”是为了避免把 10 分钟内的所有静音时间都算成 ASR 成本。
- 不同平台对音频 token、文本 token、字符数和静音段的计算方式不同，实际账单会和估算值有偏差。
- 国内组合方案中的“报告生成”统一按阿里云 `qwen-plus` 非思考模式估算，便于横向比较。这是基于官方价格做出的工程化估算。

#### 1. OpenAI 方案：全球体验优先

推荐组合：

- 实时语音对话：`gpt-realtime-2`
- 实时转写：`gpt-realtime-whisper`
- 报告生成：`gpt-5.4-mini`

官方价格口径：

- `gpt-realtime-2` 音频输入：32 美元 / 100 万 token
- `gpt-realtime-2` 音频输出：64 美元 / 100 万 token
- `gpt-realtime-2` 文本输入：4 美元 / 100 万 token
- `gpt-realtime-2` 文本输出：24 美元 / 100 万 token
- `gpt-realtime-whisper`：0.017 美元 / 分钟
- `gpt-5.4-mini`：输入 0.75 美元 / 100 万 token，输出 4.5 美元 / 100 万 token

估算方式：

- 输入音频按每 100ms 约 1 token，约 600 token / 分钟。
- 输出音频按每 50ms 约 1 token，约 1200 token / 分钟。
- 5 分钟用户语音输入：3000 audio token，成本约 0.096 美元。
- 5 分钟 AI 语音输出：6000 audio token，成本约 0.384 美元。
- 5 分钟实时转写：约 0.085 美元。
- 实时会话文本开销，按输入 3000 token、输出 1500 token 粗估：约 0.048 美元。
- 报告生成按输入 4000 token、输出 1500 token 粗估：约 0.00975 美元。

结论：

- 单次 10 分钟练习会话成本约 0.62 美元。
- 作为预算值，建议按 0.62 到 0.70 美元 / 次预估。
- 若每月 1000 次 10 分钟练习，会话模型成本可按 620 到 700 美元 / 月预估。

#### 2. 火山引擎方案：端到端实时语音，成本显著低于 OpenAI

推荐组合：

- 实时语音对话：豆包端到端实时语音大模型
- 报告生成：可继续使用豆包文本模型，或与其他低成本文本模型组合

官方价格口径：

- 输入文本：10 元 / 100 万 token
- 输入音频：80 元 / 100 万 token
- 输入文本 cached：5 元 / 100 万 token
- 输入音频 cached：5 元 / 100 万 token
- 输出文本：80 元 / 100 万 token
- 输出音频：300 元 / 100 万 token

官方 token 折算说明：

- 当前轮用户输入音频约 6.25 token / 秒。
- 其他输入音频或输出音频约 25 token / 秒。

估算方式：

- 5 分钟用户语音输入：300 秒 × 6.25 token ≈ 1875 token，成本约 0.15 元。
- 5 分钟 AI 语音输出：300 秒 × 25 token ≈ 7500 token，成本约 2.25 元。
- 实时文本开销按输入 3000 token、输出 1500 token 粗估：约 0.15 元。
- 报告生成按低成本文本模型估算：约 0.006 元，可忽略不计。

结论：

- 单次 10 分钟练习会话的基础成本约 2.41 元。
- 作为预算值，建议按 2.5 到 2.8 元 / 次预估。
- 若每月 1000 次 10 分钟练习，会话模型成本可按 2500 到 2800 元 / 月预估。

评价：

- 相比 OpenAI，运行成本显著更低。
- 相比“ASR + LLM + TTS”分段组合，接入链路更短，更适合做更像真人对话的实时语音体验。
- 但主要服务和文档生态仍偏国内，海外用户的网络可达性和跨境延迟需要单独验证。

#### 3. 阿里云方案：满足需求前提下的最低运行成本

推荐组合：

- 实时转写：Paraformer 实时语音识别
- 对话与报告生成：`qwen-plus` 非思考模式
- 实时语音输出：CosyVoice `cosyvoice-v3.5-flash`

官方价格口径：

- Paraformer ASR：0.00008 元 / 秒
- CosyVoice `cosyvoice-v3.5-flash`：0.8 元 / 万字符
- `qwen-plus`：输入 0.8 元 / 100 万 token，输出 2 元 / 100 万 token（中国内地部署）

估算方式：

- 5 分钟用户有效语音：300 秒 × 0.00008 元 = 0.024 元。
- AI 输出 1100 个汉字。按 CosyVoice 规则，汉字按 2 字符计费，即约 2200 计费字符。
- 2200 字符约为 0.22 万字符，TTS 成本约 0.176 元。
- 对话 LLM 粗估按输入 10000 token、输出 5000 token：约 0.018 元。
- 报告生成按输入 4000 token、输出 1500 token：约 0.006 元。

结论：

- 单次 10 分钟练习会话的基础成本约 0.22 元。
- 作为预算值，建议按 0.25 到 0.40 元 / 次预估。
- 若每月 1000 次 10 分钟练习，会话模型成本可按 250 到 400 元 / 月预估。

评价：

- 这是当前对比中运行成本最低的方案。
- 但它不是单模型端到端实时语音，需要自己拼接 ASR、LLM、TTS 三段链路，工程复杂度最高。
- CosyVoice 文档显示该语音合成服务当前仅在中国内地（北京）部署范围可用，因此如果直接服务海外学生，跨境网络时延和稳定性必须重点验证。

#### 4. 腾讯云方案：语音基础设施完整，但 LLM 仍需自行补齐

推荐组合：

- 实时语音对话框架：TRTC AI 实时对话
- 语音转文字：TRTC AI 智能识别
- 语音合成：TRTC TTS
- 对话与报告生成：需额外接入 LLM，本文仍按 `qwen-plus` 估算

官方价格口径：

- AI 实时对话服务：0.01 元 / 分钟
- 官方示例中的语音转文字基础版：0.05 元 / 分钟
- 官方示例中的语音合成：3 元 / 万字符
- LLM 成本不包含在腾讯云 AI 实时对话报价中，需自行采购

估算方式：

- 10 分钟 AI 实时对话服务：0.10 元。
- 若 10 分钟内用户持续开麦，则 ASR 成本约 0.50 元。
- AI 输出 1100 个汉字，按 0.11 万字符 × 3 元 / 万字符，TTS 成本约 0.33 元。
- 对话 LLM 约 0.018 元。
- 报告生成约 0.006 元。

结论：

- 单次 10 分钟练习会话成本约 0.95 元。
- 作为预算值，建议按 1.0 到 1.2 元 / 次预估。
- 若每月 1000 次 10 分钟练习，会话模型成本可按 1000 到 1200 元 / 月预估。

评价：

- 成本明显低于 OpenAI，高于阿里云组合，低于火山端到端实时语音的部分场景。
- 腾讯云实时语音基础设施成熟，但官方文档明确指出 LLM 费用需要自行采购，因此总体方案不是一体化产品。
- 对个人开发者来说，系统集成复杂度仍然不低。

#### 6. 推荐方案

如果优先级是“满足需求前提下，把国内运行成本压到最低”，推荐顺序如下：

1. 第一推荐：阿里云组合方案 `Paraformer + Qwen-Plus + CosyVoice`。
   说明：运行成本最低，适合首版 MVP 验证和小规模内测。
2. 第二推荐：火山引擎豆包端到端实时语音。
   说明：虽然运行成本高于阿里云组合，但仍显著低于 OpenAI，同时工程复杂度更低，更适合做自然实时语音对话。
3. 第三推荐：腾讯云 TRTC + 外部 LLM。
   说明：语音链路能力完整，但 LLM 不打包，方案复杂度和总体成本都不如前两者有优势。

如果优先级改为“直接服务海外中文学习者，并优先保证全球访问稳定性和上线速度”，则推荐 OpenAI 方案，接受更高的运行成本换取更低的跨境风险和更快的交付速度。

#### 7. 其他开发与运行费用

无论采用哪种模型方案，首版仍需预留以下非模型成本：

- 域名：通常约 10 到 20 美元 / 年。
- 前端托管：Vercel Pro 为 20 美元 / 月；Hobby 免费但更适合试运行。
- 后端托管：Railway 或 Render 通常需要 5 到 25 美元 / 月起步。
- PostgreSQL 与 Redis：若使用托管服务，通常还需要数美元到十几美元 / 月。
- 监控、日志、邮件通知、错误上报：正式环境通常会新增小额持续成本。

对于这个项目，模型成本通常会明显高于基础服务器费用，因此真正需要重点控制的是：

- 单次练习时长
- 用户平均轮次
- 是否保留长上下文
- 是否把报告模型与实时会话模型拆开

### 9.3 主要技术难点

#### 1. 实时链路复杂

实时语音聊天涉及浏览器录音、传输协议、服务端会话、AI 流式响应、前端同步渲染，链路比普通聊天长，调试成本高。

#### 2. 延迟与体验平衡

口语练习对停顿和响应速度敏感。识别、推理和合成任何一段过慢，都会影响“像在对话”的体验。

#### 3. 转写与展示一致性

实时转写往往先输出临时文本，再输出最终文本，前端需要处理增量修正，否则会出现重复、跳字、顺序错乱等问题。

#### 4. 报告质量控制

仅靠一次 prompt 生成的报告容易空泛。若要让报告真正有学习价值，需要设计固定的评价维度和输出模板。

#### 5. 数据模型设计

会话、消息、转写片段、报告之间需要明确关联，否则后续很难扩展历史记录、统计分析和场景练习。

### 9.4 个人开发下的风险评估

对个人开发者来说，最大风险不是代码量，而是同时处理前端音频、后端状态、AI 接口和体验调优。建议强制控制范围，避免首版同时做：登录、支付、逐词级发音评分、过多场景扩展、多端同步等扩展项。

## 10. 建议技术架构

## 10.1 架构原则

- 使用 Monorepo，统一管理前端、后端、共享类型和配置。
- 先保证核心链路跑通，再做复杂拆分。
- 尽量让实时连接直达 AI 服务，应用后端负责鉴权、会话记录和业务编排。

## 10.2 推荐 Monorepo 目录结构

```text
learn_chinese/
  apps/
    web/                 # Next.js 前端
    api/                 # NestJS 后端
  packages/
    ui/                  # 可复用 UI 组件
    types/               # 前后端共享类型
    prompts/             # 报告生成与系统提示词
    eslint-config/       # 共享 lint 配置
    tsconfig/            # 共享 tsconfig
  infra/
    docker/              # Docker 配置
    scripts/             # 部署或初始化脚本
```

## 10.3 前端建议

- 框架：Next.js
- 实时能力：浏览器麦克风采集 + WebRTC 优先，WebSocket 作为服务端流式备选
- 状态管理：Zustand 或 React Context + reducer
- UI：聊天窗口、音频权限提示、会话状态栏、报告页

前端重点处理：

- 麦克风权限
- 扬声器播放和打断控制
- 实时文本增量渲染
- 连接状态和错误提示
- 会话结束后的报告跳转

## 10.4 后端建议

- 框架：NestJS
- 能力边界：
  - 生成实时会话所需的服务端令牌或会话信息
  - 生成匿名会话 ID 或访客标识
  - 管理会话开始、结束、持久化
  - 保存聊天文本、报告结果
  - 保存场景信息与报告维度结果
  - 封装第三方 AI 接口
  - 提供报告生成接口和历史记录接口

## 10.5 数据存储建议

- PostgreSQL：保存匿名会话、消息、报告、场景配置
- Redis：保存实时会话状态、短期缓存、限流和异步任务状态
- 对象存储：MVP 不需要；仅当后续要保存原始音频时再引入

## 10.6 第三方 AI API 建议

### 方案一：OpenAI

适合目标：

- 直接面向海外中文学习者上线。
- 优先保证全球可访问性、实时稳定性和开发效率。

优点：

- 实时语音、实时转写、WebRTC 文档和全球可用性都更成熟。
- 单平台即可覆盖实时对话主链路。
- 一个人开发时，接入复杂度最低。

缺点：

- 运行成本最高。
- 若用户练习时长高、会话量上来，模型成本增长很快。

### 方案二：火山引擎豆包端到端实时语音

适合目标：

- 希望显著降低成本，同时保留端到端实时语音体验。
- 希望降低“ASR + LLM + TTS”三段式集成复杂度。

优点：

- 成本明显低于 OpenAI。
- 语音交互链路完整，更适合低延迟语音对练。
- 对中文场景更友好。

缺点：

- 海外访问稳定性和跨境时延要实测。
- 海外目标用户占比高时，网络链路风险高于 OpenAI。

### 方案三：阿里云低成本组合

推荐组合：

- ASR：Paraformer
- 对话与报告：Qwen-Plus 非思考模式
- TTS：CosyVoice `cosyvoice-v3.5-flash`

适合目标：

- 首版优先验证产品，不追求一开始就把语音体验做成最自然。
- 对运行成本极度敏感。

优点：

- 当前对比中成本最低。
- 语音识别、文本模型、语音合成都有成熟官方产品。
- 报告生成成本几乎可以忽略。

缺点：

- 需要自己拼接三段链路，工程复杂度最高。
- 语音合成能力当前地域限制更明显，海外用户直连体验需重点验证。

### 方案四：腾讯云组合

适合目标：

- 已经深度使用腾讯云生态。
- 更看重 TRTC 与实时音视频基础设施的整合能力。

优点：

- 语音链路能力完整。
- 如果后续还会做多人音视频、课堂互动、直播等场景，生态协同更自然。

缺点：

- LLM 费用不包含在官方 AI 实时对话报价中，需要额外采购。
- 单从这个项目首版来看，成本和复杂度都没有形成明显优势。

### 最终推荐

结合“目标用户主要是海外中文学习者”和“希望优先采用国内更低成本方案”这两个前提，建议分成三种决策路径：

1. 成本最低优先：选择阿里云组合方案。
   说明：这是最便宜的可落地路径，适合先做 MVP、小规模验证和封闭测试。
2. 成本与实时语音体验平衡：选择火山引擎豆包端到端实时语音。
   说明：比阿里云组合贵，但工程复杂度明显更低，也更接近完整的实时语音对话体验。
3. 海外正式上线优先：选择 OpenAI。
   说明：成本最高，但最适合直接服务海外中文学习者。

## 10.7 发布平台建议

### 推荐发布方案

针对“海外中文学习者”为主的产品定位，建议采用前后端分离部署：

- 前端发布到 Vercel。
- 后端 API 发布到 Railway 或 Render。
- PostgreSQL 与 Redis 使用托管服务。

推荐原因：

- Vercel 对 Next.js 支持最好，全球 CDN 和预览部署体验成熟，适合面向海外用户的 Web 前端。
- Railway 和 Render 更适合承载 NestJS 常驻 API、定时任务和数据库连接，不必强行把后端塞进前端平台的函数模型中。
- 如果实时语音链路采用前端直连 OpenAI WebRTC，后端负担会较轻，这种部署结构更适合首版。

### 平台选型建议

#### 方案 A：Vercel + Railway

- 适合个人开发者快速上线海外产品。
- 前端放 Vercel，后端放 Railway，部署简单，学习成本低。
- Railway 计费直观，适合从 MVP 到小规模生产的过渡。

#### 方案 B：Vercel + Render

- 适合希望把 Web API、Postgres、Redis 尽量集中在一个平台体系内管理。
- Render 对 Web 服务、数据库和 Key Value 的组合支持更完整。
- 如果后续需要更多后台任务、队列和 WebSocket 扩展，Render 的服务模型更自然。

### 不建议的首版发布方式

- 不建议首版把前端和 NestJS 后端全部压到单一 Serverless 前端平台中。
- 不建议首版自建 VPS 运维整套数据库、Redis、反向代理和监控，这会明显增加个人开发负担。

## 12. MVP 版本定义

- 多语言课程体系
- 原始音频长期存储

## 14. 结论

该项目完全可以落地，关键不在于“有没有技术路径”，而在于“首版到底把成本和体验放在哪个优先级”。

如果只看运行成本，并要求满足实时语音输入、实时语音输出、实时转写和报告生成四个核心需求，那么：

- 阿里云组合方案成本最低，适合做首版 MVP 验证。
- 火山引擎豆包端到端实时语音是成本与工程复杂度更平衡的方案。
- OpenAI 方案最适合直接面向海外中文学习者正式上线，但运行成本最高。

因此推荐的决策顺序是：

1. 若当前目标是先验证产品，优先阿里云 `Paraformer + Qwen-Plus + CosyVoice`。
2. 若当前目标是更自然的实时语音体验，且仍希望控制在国内方案成本内，优先火山引擎豆包端到端实时语音。
3. 若当前目标是直接服务海外真实用户并追求全球可访问性，优先 OpenAI。

对这个项目而言，最稳妥的工程策略不是一开始就选“技术最强”的方案，而是先确定你当前到底更在意：

- 每次练习的运行成本
- 一个人开发的实现复杂度
- 海外用户的访问稳定性

只要这个优先级排序明确，技术选型就会非常清晰。
