# history / reports 系统设计方案

## 1. 目标

本次只针对 `history` 页面和 `reports` 页面做设计，不展开代码实现。

目标是把报告详情页升级为“一个会话的完整查看页”，无论该会话是否有评分报告，用户都能看到：

- 会话基础信息
- 用户与 AI 的实际对话内容
- 若存在报告，则展示评分分析

## 2. 设计原则

- 以会话为中心，而不是以“是否有报告”为中心
- 所有历史会话都允许进入详情页
- 报告是“可选模块”，不是进入详情页的前置条件
- 页面风格保持简约，优先信息清晰和阅读流畅

## 3. 用户流程

1. 用户进入 `history` 页面
2. 历史列表中的每一条会话都可点击
3. 点击后统一跳转到 `/reports/{conversationId}`
4. 报告页先展示基础信息和对话内容
5. 如果该会话存在报告，再展示评分分析模块

## 4. 页面结构

报告详情页分为 3 个主模块，自上而下排列：

### 4.1 基础信息模块

展示本次会话的摘要信息，建议包含：

- 场景标题
- 场景类型
- 角色名称
- 难度
- 开始时间
- 结束时间
- 会话时长
- 报告状态
- 总分（如有）

说明：

- 该模块参考 `history` 列表的信息结构
- 无报告时，状态显示为 `No report` 或 `Pending`

### 4.2 对话内容模块

展示用户与 AI 的真实对话 transcript。

展示方式建议：

- 按时间顺序排列
- 区分 `User` 和 `AI`
- 一条消息一个气泡或卡片
- 用户消息靠右，AI 消息靠左
- 保留消息原文，不做改写

说明：

- 这是报告页的必备模块
- 即使没有评分报告，也必须可展示

### 4.3 分析报告模块

仅当会话存在报告时显示。

包含 4 个部分：

- Scores
- Strengths
- Issues
- Summary

明确不展示：

- Next steps

## 5. Issues 模块设计

`Issues` 不能只给抽象结论，必须包含用户说错的字、词、句法的具体纠正信息。

建议把每条 issue 设计为结构化内容：

- 原句
- 问题点
- 建议表达
- 简短说明

示例展示形态：

- Original: 用户原话
- Problem: 错字 / 错词 / 语法问题
- Better: 推荐改法
- Note: 为什么这样更自然

这样做的好处：

- 用户能直接看懂“错在哪”
- 后续也方便扩展为更细的报告能力

## 6. 接口设计

为简化前端逻辑，建议报告详情页使用“单接口聚合返回”。

### 6.1 建议接口

- `GET /reports/{conversationId}/detail`

### 6.2 返回内容

返回一个统一详情对象，包含：

- 基础信息
- transcript
- report（可为空）

建议数据结构：

```md
conversation
id
title
scenarioType
roleName
difficulty
startedAt
endedAt
durationSeconds
reportState
score

transcript[]
role
content
createdAt

report
nullable
scores
strengths[]
issues[]
summary
```

说明：

- 如果没有报告，`report = null`
- 前端只判断 `report` 是否存在，不需要多接口拼装

## 7. history 页面改造

当前 `history` 页面应统一支持所有会话跳转：

- 有评分的会话：跳转后显示完整报告页
- 无评分的会话：跳转后只显示基础信息 + 对话内容

因此，`history` 列表中的每个 item 都应具备可点击能力，不再只让“有分数的记录”可进入详情。

## 8. UI 方案

整体采用简约风格，推荐如下层次：

- 顶部：页面标题 + 返回 history
- 第一屏：基础信息卡片
- 第二屏：对话内容时间线 / 对话气泡
- 第三屏：评分分析区

视觉建议：

- 少装饰，弱化花哨背景
- 强调留白和分组
- 用轻边框、浅底色区分模块
- 分数做成简洁横向指标，不做复杂图表
- `Issues` 信息密度最高，应保证可扫描性

## 9. 状态处理

报告页需要覆盖 4 种状态：

1. 加载中
2. 会话存在但无报告
3. 会话存在且有报告
4. 会话不存在 / 加载失败

对应策略：

- 加载中：骨架屏或简单 loading
- 无报告：正常展示基础信息与 transcript，并显示 `No report available`
- 有报告：展示完整 3 大模块
- 异常：显示明确错误提示

## 10. 最小落地方案

本次建议按“最少改动、满足需求”推进：

1. history 列表所有项统一可跳转
2. reports 页增加基础信息模块
3. reports 页增加 transcript 模块
4. report 分析模块裁剪为 `Scores / Strengths / Issues / Summary`
5. `Issues` 输出改为更具体的纠错内容
6. 无报告场景支持正常渲染

## 11. 验收标准

- 所有历史会话都能进入 `/reports/{conversationId}`
- 无报告会话也能看到基础信息和实际对话
- 有报告会话能额外看到评分分析
- 分析区只包含 `Scores / Strengths / Issues / Summary`
- `Issues` 中能看到具体纠错信息，而不是只有抽象评价
- 页面整体风格简约、结构清晰、阅读顺序自然
