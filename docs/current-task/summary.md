# Task Summary

## 目标

- 实现用户可以通过真实豆包端到端实时语音进行中文对话练习。
- 将当前内存态会话和报告持久化到 PostgreSQL + Redis。

## 当前状态

- `apps/web` 已有首页、practice、history、report 等页面骨架。
- practice 页面支持会话创建、transcript 展示、开始/暂停/结束操作，但未接入真实豆包实时语音。
- `apps/api` 已有 `realtime`、`conversation`、`report`、`history` 等接口骨架，但数据仍保存在内存中。
- 实时语音当前是浏览器本地 `SpeechRecognition` / `speechSynthesis` 方案，不是真正的豆包实时语音。

## 现状缺口

- 未接入真实豆包端到端实时语音链路。
- 未使用 PostgreSQL 持久化会话、报告、历史数据。
- 未使用 Redis 管理实时状态和任务状态。
- 当前对话仍是“点说一句”的浏览器语音模式，不是持续实时交互。
- 服务重启后会话数据会丢失。

## 核心方案

- 前端：接入火山引擎 RTC 实时语音 SDK，建立真正的实时语音会话。
- 后端：提供会话令牌、业务路由、历史与报告接口，并负责持久化写入 PostgreSQL。
- Redis：保存短期会话状态、任务状态、幂等控制和实时运行态数据。
- 设计应避免把浏览器本地语音识别当作最终方案，仅作为临时兜底。

## 关键工作

- 真实会话链路：`GET /api/session/start` / `POST /api/realtime/session` 等接口，返回豆包/RTC 鉴权参数。
- 数据库持久化：会话、消息、报告、场景、用户历史写入 PostgreSQL。
- 运行态管理：会话状态、转写状态、实时任务状态写入 Redis。
- 区分会话阶段：对话前可选角色，对话中禁止切换角色。
- 文案要求：练习主题英文，页面文案英文，只有实时对话问题可以展示中文。

## 验收标准

- 首页点击主题进入 `practice`，可以发起真实豆包实时语音对话。
- 实时对话过程真实可用，不依赖浏览器本地语音识别模拟。
- 会话与报告数据写入 PostgreSQL，重启后可查询历史。
- Redis 用于实时状态和短期运行态，不作为最终持久化存储。
- 整体页面和文案保持英文为主，符合海外学生使用场景。

## 当前优先级

1. 真实豆包实时语音链路接入。
2. PostgreSQL 持久化会话与报告。
3. Redis 运行态状态管理。
4. practice 页面流程与角色逻辑完善。
5. history/report 查询与展示。
